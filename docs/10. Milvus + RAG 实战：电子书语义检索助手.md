# Milvus + RAG 实战：电子书语义检索助手 - 总结

对应代码：课程仓库 [ai-agent-course-code](https://github.com/QuarkGluonPlasma/ai-agent-course-code) 内 milvus-test 项目下的电子书示例（如 `src/ebook-writer.mjs`、`src/ebook-query.mjs`、`src/ebook-reader-rag.mjs`）。环境变量与第 6、7、9 章一致；依赖需包含 `@zilliz/milvus2-sdk-node`、`@langchain/openai`、`@langchain/community`（EPubLoader）、`@langchain/textsplitters`、`epub2`、`html-to-text`、`dotenv`。需先启动 Milvus（Docker），并准备 .epub 文件（示例为《天龙八部》）。

---

## 一、实战目标

- 将 **Loader、Splitter、Milvus** 串成完整 RAG：用 loader 从来源加载文档 → splitter 分块 → 嵌入模型向量化 → 存入 Milvus；查询时把 query 向量化，按余弦相似度检索最相关片段，再交给大模型生成回答。
- 本节做**电子书语义检索助手**：对 .epub 电子书（如《天龙八部》）建向量库，用户用自然语言问「段誉会什么武功」「鸠摩智会什么武功」等，系统做语义检索后由大模型基于原文回答。这类问题无法用 MySQL 关键词查，必须用向量库 + RAG。

## 二、电子书集合 Schema 与建库思路

- **Collection**：`ebook_collection`；若不存在则 **hasCollection → createCollection → createIndex（vector）→ loadCollection**。
- **Schema 要点**：`id`（主键，如 `bookId_chapterNum_index`）、`book_id`（与 MySQL 的 book 表关联，后续联动用）、`book_name`、`chapter_num`（第几章）、`index`（章内第几个块）、`content`（文本）、`vector`（FloatVector，维度与嵌入模型一致，如 1024）。vector 做语义匹配，其余为元信息。
- **写入流程**：EPubLoader 按章加载 → 每章再用 RecursiveCharacterTextSplitter 按 500 字分块（chunkOverlap 50）→ 每块向量化后 insert 入 Milvus；可逐章流式处理，边拆边插。

## 三、ebook-writer.mjs：建库与写入

| 步骤 | 说明 |
|------|------|
| 连接与集合 | 连接 Milvus，**ensureCollection(bookId)**：hasCollection 为 false 时 createCollection（含上述 schema）、createIndex（IVF_FLAT + COSINE）、再 loadCollection。 |
| 加载 EPUB | **EPubLoader(EPUB_FILE, { splitChapters: true })**，**loader.load()** 得到按章划分的 Document 数组。 |
| 二次分块 | **RecursiveCharacterTextSplitter({ chunkSize: 500, chunkOverlap: 50 })**，对每章 pageContent 做 **splitText**，得到多个文本块。 |
| 向量化与插入 | 对每个块 **getEmbedding(chunk)**，构造含 id、book_id、book_name、chapter_num、index、content、vector 的条目，**client.insert** 批量写入；按章循环，可输出进度与累计条数。 |

- 整本书会生成数千条记录（如《天龙八部》3000+），每条带章节与块索引，便于回溯；book_id 预留与 MySQL 联动。

## 四、ebook-query.mjs：向量检索

- **loadCollection** 确保集合已加载（若已加载可忽略报错）。
- 将用户 query（如「段誉会什么武功？」）**getEmbedding** 得到向量，**client.search** 指定 collection_name、vector、limit、metric_type: COSINE、output_fields（如 id、book_id、chapter_num、index、content），得到最相似的若干条及 score。
- 语义检索可命中未含关键词但语义相关的片段（如「北冥神功」「凌波微步」），MySQL 关键词无法等价实现。

## 五、ebook-reader-rag.mjs：完整 RAG 问答

- **检索**：**retrieveRelevantContent(question, k)**：问题向量化 → Milvus search → 返回 k 条最相关片段（含 score、chapter_num、content 等）。
- **上下文**：将检索结果格式化为「片段 i / 章节 / 内容」，用分隔符拼成一大段 context。
- **Prompt**：声明角色为《天龙八部》助手，要求基于给定片段回答、可综合多片段、可引用原文、无相关信息则如实告知；把 context 与用户问题一并放入 prompt。
- **生成**：**ChatOpenAI.invoke(prompt)** 得到最终回答；未做流式时需等待生成完成。

## 六、依赖与环境

- **依赖**：`@zilliz/milvus2-sdk-node`、`@langchain/openai`、`@langchain/community`（EPubLoader 在 `document_loaders/fs/epub`）、`@langchain/textsplitters`、`epub2`、`html-to-text`、`dotenv`。
- **环境变量**：与第 6、7、9 章相同（OPENAI_API_KEY、OPENAI_BASE_URL、MODEL_NAME、EMBEDDINGS_MODEL_NAME）。
- **运行顺序**：先运行 ebook-writer 建库并写入电子书；再运行 ebook-query 做纯检索验证，或 ebook-reader-rag 做问答。

## 七、总结

- **实战链路**：EPubLoader 按章加载 → RecursiveCharacterTextSplitter 按 500 字分块 → 嵌入向量化 → 写入 Milvus（ebook_collection）→ 查询时 question 向量化 → Milvus 检索 top-k 片段 → 拼入 prompt → 大模型基于原文回答；与公司内文档/知识库 RAG 流程一致，仅数据源换为 epub。
- **Schema**：id、book_id、book_name、chapter_num、index、content、vector；book_id 用于与 MySQL book 表关联，后续可做「Milvus 检索 + MySQL 联表」的联动。
- **扩展**：将 epub 换成内部 PDF、Word、网页等，用对应 Loader + 同一套 Splitter/Milvus/RAG 即可实现企业文档语义检索与问答。
