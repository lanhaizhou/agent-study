# 知识库的 loader 和 splitter：从各种来源加载文档并分割成小块 - 总结

对应代码：`chapter/7/rag-test/src/loader-and-splitter.mjs`。环境变量与第 6 章一致：根目录 `.env` 配置 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`MODEL_NAME`、`EMBEDDINGS_MODEL_NAME`；依赖需包含 `@langchain/community`（Loader）、`@langchain/textsplitters`（Splitter）、`cheerio`（网页解析）。

---

## 一、为什么需要 Loader

- 第 6 章示例中直接手写 **Document** 作为知识库；实际场景里知识来源多样：Word、PDF、网页 URL、YouTube、社交媒体等，无法全部手写。
- **Loader** 的作用：从各类来源（文件、URL、API 等）读取原始内容，统一转成 **Document**（pageContent + metadata），供后续向量化与检索使用。

## 二、Loader 概览

- **含义**：按「数据源类型」封装的加载器，输入为源地址或配置，输出为 **Document** 或 **Document[]**。
- **常见来源**：本地文件（PDF、Word、TXT）、网页（HTML）、视频/音频文稿、数据库、API 等；LangChain 文档中列出的 Document Loader 有 180+，多为社区维护。
- **包位置**：Loader 多在 **@langchain/community** 下，按来源分子包（如 `document_loaders/web/cheerio`）。
- **示例**：**CheerioWebBaseLoader** 请求指定 URL，用 Cheerio 解析 HTML，通过 **selector**（如 `.main-area p`）抽取 DOM 中的文本，合并为一个或多个 Document；适合抓取网页正文做 RAG。

## 三、为什么需要 Splitter

- 单篇文档可能很长（如一本书、一篇长文），若整篇作为一个 Document 向量化，单向量难以表达局部语义，检索时也容易返回整篇、上下文过长。
- **Splitter（文本分割器）**：将长文档按一定规则切分成多个**小块**，每块仍为 **Document**，再分别向量化入库；检索时返回与问题最相关的若干**块**，控制上下文粒度与长度。

## 四、Splitter 要点（RecursiveCharacterTextSplitter）

- **块大小**：**chunkSize** 设定每块的目标字符数上限；过长则继续在块内按分隔符切。
- **块重叠**：**chunkOverlap** 表示相邻块之间重叠的字符数，避免句子在块边界被截断、丢失上下文。
- **分隔符**：**separators** 为按优先级尝试的分割符列表（如 `["。", "！", "？"]`），优先在句号、问号等边界切分，保持语义完整；若某级分隔符切完仍超 chunkSize，会递归使用下一级（如换行、空格）。
- **包位置**：**@langchain/textsplitters**；常用 **RecursiveCharacterTextSplitter**，兼顾「按语义边界切」与「控制块大小」。

## 五、代码流程与要点（loader-and-splitter.mjs）

| 步骤          | 说明                                                                                                                                                                                                      |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Loader 加载   | **CheerioWebBaseLoader(url, { selector })** 创建加载器，**loader.load()** 得到 **Document[]**；本例 selector 取网页主区域段落，得到一大段 pageContent。                                                   |
| Splitter 切分 | **RecursiveCharacterTextSplitter({ chunkSize, chunkOverlap, separators })** 创建分割器，**splitter.splitDocuments(documents)** 将 Document 切为多个小块 Document，metadata 会继承。                       |
| 向量化与检索  | **MemoryVectorStore.fromDocuments(splitDocuments, embeddings)** 将切分后的块向量化入库；**asRetriever({ k })** 得到检索器；后续与第 6 章一致：retriever.invoke(question) → 拼入 prompt → 大模型生成回答。 |

- 整体链路：**原始来源 → Loader → Document(s) → Splitter → 小块 Document[] → 向量库 → 检索 → 增强 prompt → 生成**。

## 六、依赖与环境

- **依赖**：`@langchain/openai`、`@langchain/core`、`@langchain/classic`（向量库）、**@langchain/community**（Loader）、**@langchain/textsplitters**（Splitter）、**cheerio**（CheerioWebBaseLoader 解析 HTML）、`dotenv`。
- **环境变量**：与第 6 章相同（OPENAI\_\*、MODEL_NAME、EMBEDDINGS_MODEL_NAME）；无额外配置。
- 可将 URL、selector、chunkSize、chunkOverlap、separators 等改为参数或配置文件，以适配不同网页与文档类型。

## 七、总结

- **Loader**：从各种来源（网页、PDF、Word、API 等）加载内容并转为 **Document**，多在 **@langchain/community**；示例中用 **CheerioWebBaseLoader** 按 selector 抓取网页正文。
- **Splitter**：将大 Document 按 **chunkSize**、**chunkOverlap**、**separators** 切为小块 Document，便于向量化与检索时控制粒度；常用 **RecursiveCharacterTextSplitter**，在 **@langchain/textsplitters**。
- **流程**：Loader 产出 Document → Splitter 切块 → 小块 Document 向量化入库 → 检索最相关块 → 拼入 prompt → 大模型回答；与第 6 章 RAG 流程一致，只是「文档从哪来、如何切」由 Loader 和 Splitter 承担。
- **扩展**：换用其他 Loader（如 PDF、Markdown、数据库）和不同 Splitter 参数，即可支持多源、多格式知识库的 RAG 构建。
