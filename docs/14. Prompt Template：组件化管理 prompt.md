# Prompt Template：组件化管理 prompt - 总结

---

## 一、这节在解决什么问题

- Prompt 是 Agent 的核心，RAG 检索结果等也会塞进 prompt；工程上更关心 **如何管理、复用、组合**，而不是「文案怎么写」（常由产品/业务负责）。
- 避免到处 **字符串拼接**：用 LangChain 的 **Prompt Template API** 做占位符、分块、对话结构、示例注入与动态选例。

## 二、`PromptTemplate`：单段字符串模板

- **`PromptTemplate.fromTemplate(...)`**（传入模板字符串）：字符串里用 **`{变量名}`** 占位。
- **`await template.format({ ... })`**：填入变量，得到 **一整段字符串**，再交给 `model.stream(prompt)` 等。
- 同一模板可换不同 `format` 入参，复用同一套结构与约束。

## 三、`PipelinePromptTemplate`：多块拼成一大块

- 把 prompt 按 **人设、背景、任务、格式** 等拆成多个子 `PromptTemplate`，各自维护。
- **`new PipelinePromptTemplate({ pipelinePrompts: [{ name, prompt }, ...], finalPrompt })`**：`finalPrompt` 里用 `{persona_block}` 等占位符引用各子块输出。
- **好处**：块可单独改、**跨场景复用**（例如周报与季度 OKR 邮件共用「人设 + 背景」）。
- `inputVariables` 可手写，但占位符也会自动识别，不必过度依赖手写列表。

## 四、`partial`：预填固定变量

- **`await pipelinePrompt.partial({ company_name, tone, ... })`**：把长期不变的字段先绑死，后续 **`format` 只传变化部分**（团队、周期、活动等），减少重复与出错。

## 五、`ChatPromptTemplate`：面向 Chat 的 messages

- 实际调用多为 **System / Human / AI / Tool** 消息数组，用 **`ChatPromptTemplate.fromMessages([['system', '...'], ['human', '...'], ...])`**。
- **`formatMessages({...})`** 得到消息列表，再 **`model.invoke(chatMessages)`**。
- 等价写法：**`SystemMessagePromptTemplate` / `HumanMessagePromptTemplate.fromTemplate`** 再 `fromMessages([systemTemplate, humanTemplate])`，语义更清晰。

## 六、Pipeline 的最终产物也可以是 Chat

- **`finalPrompt` 可以是 `ChatPromptTemplate`**，前面各块仍是 `PromptTemplate` 拼进某条 human/system 的正文里。
- 用 **`formatPromptValue({...})`**，再 **`.toChatMessages()`** 得到最终消息数组（与单纯字符串 `format` 区分）。

## 七、`MessagesPlaceholder`：插入多轮历史

- 在 `fromMessages` 里放 **`new MessagesPlaceholder('history')`**，**`formatPromptValue({ history: [...], current_input })`** 把历史对话插到固定槽位。
- 适合「系统指令 + **可变长度对话** + 当前问题」类模板。

## 八、Few-shot：固定示例与动态选例

| 能力 | 作用 |
|------|------|
| **`FewShotPromptTemplate`** | `examplePrompt` + `examples` + `prefix`/`suffix`，拼出「带少量示例」的字符串 prompt。 |
| **`FewShotChatMessagePromptTemplate`** | 示例是 **human/ai 多轮片段**，嵌进 `ChatPromptTemplate`，与「纯文本 few-shot」对应。 |
| **`LengthBasedExampleSelector`** | 按 **剩余长度预算**（`maxLength`、可自定义 `getTextLength` 近似 token）从示例池里选子集，控制 token。 |
| **`SemanticSimilarityExampleSelector`** | 按 **与当前 query 的语义相似度** 选例；可用 **Milvus** 存示例向量，`Milvus.fromExistingCollection` + `k` 取 Top-K。 |

- 示例很多或需按 query 变化时：**不要全量塞进 prompt**，用 **ExampleSelector** 只选最相关或长度允许的几条。

## 九、依赖与扩展（课程原文涉及）

- 基础：**`dotenv`、`@langchain/core`、`@langchain/openai`**。
- 语义选例 + Milvus：**`@langchain/community`、`@zilliz/milvus2-sdk-node`**，以及嵌入模型相关环境变量（与前面 Milvus/RAG 章节一致）。

## 十、总结

- **组件化思路**：**`PromptTemplate`** 管单段；**`PipelinePromptTemplate`** 管多块组合与复用；**`partial`** 管默认值。
- **对话形态**：**`ChatPromptTemplate`**（及 System/Human 子模板）+ **`MessagesPlaceholder`** 管历史；Pipeline 的终点也可接到 Chat，用 **`formatPromptValue` → `toChatMessages()`**。
- **示例策略**：**`FewShotPromptTemplate` / `FewShotChatMessagePromptTemplate`** 注入风格与结构；示例多时配合 **`LengthBasedExampleSelector`** 或 **`SemanticSimilarityExampleSelector`（向量库）** 控制成本与相关性。
- **实践建议**：API 多但概念线性，过一遍知道有哪些能力即可，需要时再查文档深入；上下文膨胀问题交给 **Memory** 等章节。
