# LangChain 全部 Splitter，其实只需要其中的一个 - 总结

对应代码：课程仓库 [ai-agent-course-code](https://github.com/QuarkGluonPlasma/ai-agent-course-code) 内 splitter 相关示例（如 CharacterTextSplitter-test.mjs、RecursiveCharacterTextSplitter-test.mjs、TokenTextSplitter-test.mjs、recursive-splitter-markdown.mjs、recursive-splitter-latex.mjs、recursive-splitter-code.mjs）。环境变量与第 6、7 章一致；依赖需包含 `@langchain/textsplitters`、`@langchain/core`（Document）；若需按 token 计算长度或验证 chunk 的 token 数，可安装 `js-tiktoken`。

---

## 一、为什么要把所有 Splitter 过一遍

- 第 7 章已用 **RecursiveCharacterTextSplitter** 完成「Loader → Document → Splitter 切块 → 向量化」流程；LangChain 在 `@langchain/textsplitters` 中还提供多种 Splitter，容易让人纠结选哪个。
- 本节梳理各 Splitter 的继承关系与适用场景，结论是：**绝大多数场景直接用 RecursiveCharacterTextSplitter 即可**，其余多为它的子类或可被其替代。

## 二、核心概念：separator、chunk size、overlap

- **separator（分隔符）**：按何种边界切分文本（如句号、换行、逗号）。先按 separator 切，再按 chunk size 装进各个 chunk。
- **chunk size**：单块目标大小上限（默认按**字符数**计）；若某段按当前 separator 切完后仍超长，会尝试用列表中的下一个 separator 继续切，形成递归分割。
- **chunk overlap**：相邻块之间重叠的字符数；**仅在文本因超长被截断时**才会在断点处加入 overlap，不是每块都有。通常设为 chunkSize 的 10%–20%，用少量重复换取语义连贯。

## 三、LangChain Splitter 继承关系与特点

- 所有 Splitter 均继承自 **TextSplitter**；常用几种关系与特点如下。

| Splitter                           | 特点                                                                                                                | 不足 / 建议                                                                                                                     |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------ | ----------------------------------------------------------- |
| **CharacterTextSplitter**          | 只支持**单一**分隔符（如 `\n` 或 `。`），按该符切分后装块。                                                         | 超长段落不会再用其他分隔符继续切，容易产生超出 chunk size 的大块；一般不推荐。                                                  |
| **RecursiveCharacterTextSplitter** | 支持 **separators 数组**（如 `['\n', '。', '，']`），按顺序尝试分隔符，切到满足 chunk size 为止；优先保证语义边界。 | 无；通用场景首选。                                                                                                              |
| **TokenTextSplitter**              | 按 **token 数**切块（需指定 encodingName，如 `cl100k_base`），便于精确控制送入模型的 token 量。                     | 为凑满 token 数会强行截断，易破坏语义；需按 token 控制时，更推荐用 RecursiveCharacterTextSplitter + 自定义 **lengthFunction**。 |
| **MarkdownTextSplitter**           | 按 Markdown 标题层级（#、##、### 等）递归切分，内置分隔符。                                                         | 本质是 RecursiveCharacterTextSplitter 的封装，仅对 Markdown 内容更方便。                                                        |
| **LatexTextSplitter**              | 按 LaTeX 语法结构切分公式与段落。                                                                                   | 同上，为 RecursiveCharacterTextSplitter 的封装。                                                                                |
| **代码分割**                       | \*\*RecursiveCharacterTextSplitter.fromLanguage('js'                                                                | 'python'                                                                                                                        | ...)\*\*，按语言语法切分，不破坏函数、类等结构。 | 支持 java、go、js、html、python、rust、swift、markdown 等。 |

- **字符与 token**：按字符设 chunk size 时，每块对应的 token 数不固定（与模型分词器有关）；若要严格按 token 控制（如计费、上下文长度），可用 **lengthFunction** 把「长度」定义为 token 数（例如用 `js-tiktoken` 的 `enc.encode(text).length`），仍用 RecursiveCharacterTextSplitter 的递归分割逻辑。

## 四、代码要点（与示例对应）

| 步骤                           | 说明                                                                                                                                                                          |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| CharacterTextSplitter          | **CharacterTextSplitter({ separator, chunkSize, chunkOverlap })**；单一分隔符，超长不继续切。                                                                                 |
| RecursiveCharacterTextSplitter | **RecursiveCharacterTextSplitter({ chunkSize, chunkOverlap, separators })**；多分隔符递归切分；可选 **lengthFunction: (text) => enc.encode(text).length** 改为按 token 计长。 |
| TokenTextSplitter              | **TokenTextSplitter({ chunkSize, chunkOverlap, encodingName })**；按 token 数切块，会强行截断，一般不推荐。                                                                   |
| Markdown / LaTeX               | **MarkdownTextSplitter**、**LatexTextSplitter** 用法同 RecursiveCharacterTextSplitter，只需 chunkSize、chunkOverlap，分隔符内置。                                             |
| 代码                           | **RecursiveCharacterTextSplitter.fromLanguage('js', { chunkSize, chunkOverlap })**，按语言返回对应 separators 的实例。                                                        |

- 验证 chunk 的字符长与 token 数：用 **js-tiktoken** 的 `getEncoding("cl100k_base").encode(text).length` 即可。

## 五、依赖与环境

- **依赖**：`@langchain/textsplitters`、`@langchain/core`（Document）；可选 `js-tiktoken`（OpenAI 系分词，用于 lengthFunction 或验证 token 数）。
- **环境变量**：与第 6、7 章一致（OPENAI\_\*、MODEL_NAME、EMBEDDINGS_MODEL_NAME）；本节示例不强制依赖嵌入或大模型，仅做切分与打印。

## 六、总结

- **结论**：直接用 **RecursiveCharacterTextSplitter** 即可；CharacterTextSplitter 太死板，TokenTextSplitter 易破坏语义，Markdown/Latex/代码分割均为 RecursiveCharacterTextSplitter 的封装或 **fromLanguage** 用法。
- **分割逻辑**：先按 **separators** 顺序切分，再按 **chunk size** 装块；超长则用下一级 separator 继续切，断点处加 **overlap**；默认按字符计长，需按 token 时用 **lengthFunction** 配合 js-tiktoken。
- **代码场景**：用 **RecursiveCharacterTextSplitter.fromLanguage('js' | 'python' | ...)** 按语法切分，避免破坏代码结构。
- **扩展**：与第 7 章结合，Loader 产出 Document → 用 RecursiveCharacterTextSplitter（或 Markdown/Latex/fromLanguage）切块 → 向量化入库 → 检索增强生成，即完整 RAG 文档准备链路。
