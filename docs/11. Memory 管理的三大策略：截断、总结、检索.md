# Memory 管理的三大策略：截断、总结、检索 - 总结

---

## 一、为什么需要 Memory 管理

- 大模型**无状态**：单次调用不记得上次对话；能「接着聊」是因为我们把历史放进 **messages**（SystemMessage、HumanMessage、AIMessage、ToolMessage 等）再一起传给模型，这就是 **Memory**。
- 若不做管理，messages 不断追加，迟早超出模型**上下文长度上限**（如 200k token），因此需要 Memory 管理策略。

## 二、存储层：ChatMessageHistory

- **作用**：决定 messages **存哪里**，与「截断、总结、检索」等逻辑解耦。
- **InMemoryChatMessageHistory**：存内存，进程结束即丢失；**addMessage**、**getMessages**、**clear** 等。
- **FileSystemChatMessageHistory**：按 **sessionId** 持久化到指定 **filePath**（如 `chat_history.json`），新进程用同一 filePath + sessionId 可**恢复历史**并继续对话，即长时记忆（LTM）；Cursor、Claude 的对话持久化即此类。
- 旧版 BufferMemory、SummaryMemory 等「逻辑层」API 已废弃（迁至 @langchain/classic），官方建议用 **history + 自己实现截断/总结/检索**；仅保留 **trimMessages** 用于按 token 截断。

## 三、策略一：截断（Truncation）

- **思路**：超出一定**条数**或**token 数**时，只保留**最近**的若干条/若干 token，丢弃更早的。
- **按条数**：对 `history.getMessages()` 的结果做 **slice(-maxMessages)**，保留最近 maxMessages 条。
- **按 token**：使用 **trimMessages(messages, { maxTokens, tokenCounter, strategy: "last" })**；**tokenCounter** 可用 **js-tiktoken** 对每条消息 content 编码后累加长度，从而在不超过 maxTokens 的前提下保留最近的消息。
- 截断后的 messages 再拼上 SystemMessage 等传给模型即可。

## 四、策略二：总结（Summarization）

- **思路**：当消息条数或 token 数超过阈值时，用**大模型**对「要丢弃的旧消息」生成**摘要**，然后清空或截断历史，仅保留「摘要（可写入一条系统或用户消息）+ 最近几条原始消息」；后续对话基于「摘要 + 最近消息」继续，节省 token。Cursor、Claude 达到上下文限制时自动触发总结即此类。
- **实现要点**：用 **getBufferString(messages, { humanPrefix, aiPrefix })** 把 HumanMessage/AIMessage 格式化成「用户: … 助手: …」的对话文本，拼成 prompt 让 LLM 输出总结；根据阈值（条数或 token）决定哪些消息参与总结、保留最近几条；**history.clear()** 后只 **addMessage** 保留的最近消息（可选把总结也塞进一条消息）。
- **按 token 触发**：用 js-tiktoken 统计总 token，超过 maxTokens 时从后往前累加消息直至达到 keepRecentTokens，前面部分送去总结，后面部分保留。

## 五、策略三：检索（Retrieval）

- **思路**：把**历史对话**（每轮或每若干轮）格式化成文本，用嵌入模型向量化后存入 **Milvus**；用户新问题时，将问题向量化，在 Milvus 中做**相似度检索**，取最相关的若干条历史对话拼进 prompt，再调用大模型回答，即对「对话历史」做 RAG。
- **写入**：Collection 如 `conversations`，字段含 id、vector、content（如「用户: … 助手: …」）、round、timestamp 等；每轮或每 N 轮对话结束后，对 content 做 **embedQuery** 后 **insert**。
- **检索**：用户输入 → **getEmbedding** → **client.search**（collection_name、vector、limit、metric_type: COSINE、output_fields）→ 得到最相关历史条及 score，拼成上下文写入 prompt → **model.invoke**；回答后可将本轮对话再 insert 进 Milvus，供后续检索。

## 六、代码要点与流程

| 模块         | 说明                                                                                                                                                                                                                  |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| History 存储 | **InMemoryChatMessageHistory()** 或 **FileSystemChatMessageHistory({ filePath, sessionId })**；**addMessage** / **getMessages** / **clear**。                                                                         |
| 截断         | 条数：**getMessages()** 后 **slice(-maxMessages)**；token：**trimMessages(messages, { maxTokens, tokenCounter, strategy: "last" })**，tokenCounter 用 js-tiktoken 计长。                                              |
| 总结         | **getBufferString(messages)** 格式化 → 用 ChatOpenAI 生成总结 prompt → 保留最近 N 条或最近 M token，其余替换为总结；**clear** 后只 addMessage 保留部分（+ 可选总结消息）。                                            |
| 检索         | Milvus 建表 conversations（id、vector、content、round、timestamp）→ 每轮/每 N 轮 insert 向量化 content；查询时 **retrieveRelevantConversations(query, k)** → 拼入 prompt → invoke；可选每轮回答后再 insert 当前对话。 |

- 三种策略可组合：例如每聊 20 条触发一次总结，将**摘要**写入 Milvus；用户问「我们聊过什么」时，从 Milvus 检索摘要再生成回答。

## 七、总结

- **Memory**：通过管理传给模型的 **messages**（存储 + 截断/总结/检索），让大模型能在多轮对话中「记得」之前的内容；存储层用 **ChatMessageHistory**（内存或文件等），逻辑层可自行实现三种策略。
- **截断**：按条数或 token 只保留最近消息，**trimMessages** 支持按 token 截断并需自定义 tokenCounter。
- **总结**：超过阈值时用 LLM 对旧消息生成摘要，只保留摘要 + 最近消息，减少 token 占用；Cursor/Claude 的自动总结即此策略。
- **检索**：历史对话向量化存入 Milvus，新问题时 RAG 检索相关历史再拼入 prompt，适合长跨度、按语义回忆；可与总结结合（如摘要存 Milvus，问「聊过什么」时检索摘要）。
- **依赖与环境**：`@langchain/core`、`@langchain/openai`、`@langchain/community`（FileSystemChatMessageHistory）、`js-tiktoken`、`dotenv`；检索需 `@zilliz/milvus2-sdk-node` 与 Milvus。环境变量与前面章节一致。
- **扩展**：管理好 Memory 后，可随时基于历史话题继续聊，是 AI Agent 多轮对话与长时记忆的基础。
