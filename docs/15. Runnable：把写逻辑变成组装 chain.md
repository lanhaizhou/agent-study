# Runnable：把写逻辑变成组装 chain - 总结

---

## 一、这节在解决什么问题

- LangChain 里 **PromptTemplate、OutputParser、ChatOpenAI** 等大量 API 都实现了 **Runnable** 接口。
- 核心转变：从 **命令式**（逐步 `format` → `invoke` → `parse`）变成 **声明式**（组装 chain，统一 `invoke` / `stream` / `batch`）。
- 这种声明式组合方式叫 **LCEL**（LangChain Expression Language）：用实现了 Runnable 的组件拼成链，由框架负责调用顺序与能力透传。

## 二、改造前后对比

| 写法       | 特点                                                                                                                              |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------- |
| **改造前** | 手动 `promptTemplate.format` → `model.invoke` → `outputParser.invoke`，三步逻辑散落。                                             |
| **改造后** | `RunnableSequence.from([promptTemplate, model, outputParser])` 或 `prompt.pipe(model).pipe(parser)`，一次 `chain.invoke(input)`。 |

- `pipe` 与 `RunnableSequence.from` **本质相同**（`pipe` 源码也返回 `RunnableSequence`）。
- 输入仍可为对象，例如 `{ text, format_instructions: outputParser.getFormatInstructions() }`。

## 三、Runnable 的统一能力
****
- 链上每个组件都实现 **`invoke`、`stream`、`batch`**。
- 对整条 chain 调用时，会 **依次** 调用各组件的对应方法：
  - **invoke**：同步单次执行。
  - **batch**：并发执行多次独立 invoke。
  - **stream**：沿链流式返回（各组件支持 stream 时自然串联）。
- 好处：组装好 chain 后，不必再关心内部如何逐步调用。

## 四、常用 Runnable API

| API                            | 作用                                                                                                        |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------- |
| **RunnableSequence**           | 顺序执行多个 Runnable（或用 `pipe` 串联）。                                                                 |
| **RunnableLambda**             | 把普通函数包装成 Runnable，可插入 chain。                                                                   |
| **RunnableMap**                | **并行**执行多个 Runnable，同一 `input` 分别处理，结果合并为对象的各属性。                                  |
| **RunnableBranch**             | **if / else if / else**：`[[条件, 分支], ..., 默认分支]`，按条件选分支执行。                                |
| **RouterRunnable**             | **switch case**：`{ key, input }` 按 `key` 路由到 `runnables` 中对应 chain。                                |
| **RunnablePassthrough**        | 透传原始输入；**`assign`** 在保留原属性的基础上合并新属性（类似 `Object.assign`）。                         |
| **RunnableEach**               | 对 **数组** 每个元素应用同一条子 chain（循环 map）。                                                        |
| **RunnablePick**               | 从对象中 **挑选指定字段** 返回。                                                                            |
| **RunnableWithMessageHistory** | 给 chain 加 **对话记忆**：按 `sessionId` 管理 `ChatMessageHistory`，配合 `MessagesPlaceholder("history")`。 |

## 五、LCEL 语法糖

- 在 chain 里可直接写 **函数** → 自动转为 `RunnableLambda`。
- 可直接写 **对象** → 自动转为 `RunnableMap`。
- `RunnablePassthrough.assign({ ... })` 适合「保留上游字段 + 扩展新字段」的场景。

## 六、RunnableWithMessageHistory 要点

- 用 `ChatPromptTemplate` + `MessagesPlaceholder("history")` 插入历史消息。
- `getMessageHistory(sessionId)` 管理各会话的 `InMemoryChatMessageHistory`（也可换数据库存储）。
- 配置 `inputMessagesKey`（如 `question`）、`historyMessagesKey`（如 `history`）。
- 调用时通过 **`configurable: { sessionId }`** 区分会话，实现多轮记忆。

## 七、心智模型

- **命令式 → 声明式**：类似从 jQuery 操作 DOM 到 Vue/React 写模板——框架负责「怎么调用」，你只关心「链怎么拼」。
- 与 **RxJS 管道**、**compose** 类似：零散逻辑组装成复合单元，统一入口。
- LCEL 是 **底层机制**；更复杂的图编排可用 LangGraph，二者可配合而非完全替代。

## 八、依赖与环境

- 依赖：`dotenv`、`@langchain/core`、`@langchain/openai`、`zod`。
- 环境变量：`OPENAI_API_KEY`、`OPENAI_BASE_URL`、`MODEL_NAME` 等（与前面章节一致）。

## 九、总结

- **Runnable** 是 LangChain 的组合抽象：组件实现同一接口，用 **LCEL** 声明式拼 chain。
- **RunnableSequence / pipe** 负责串联；**RunnableMap** 负责并行；**RunnableBranch / RouterRunnable** 负责分支与路由。
- **RunnableLambda、RunnablePassthrough、RunnableEach、RunnablePick** 负责函数封装、透传、批处理与字段选取。
- **RunnableWithMessageHistory** 在 chain 外挂会话级 memory。
- 整条 chain 统一支持 **invoke / stream / batch**，写 Agent 时优先「组装 chain」，而不是手写逐步调用。
