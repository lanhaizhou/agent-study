# Output Parser 实战：智能录入 + 流式版 mini cursor - 总结

对应代码：课程仓库 [ai-agent-course-code](https://github.com/QuarkGluonPlasma/ai-agent-course-code) 内 **output-parser-test** 项目（如 `src/test/create-table.mjs`、`src/test/smart-import.mjs`、`src/structured-json-schema.mjs`、`src/test/mini-cursor.mjs`）。流式 Agent 所用工具定义可从 **tool-test** 项目复制 `all-tools.mjs` 等。环境变量：根目录 `.env` 配置 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`MODEL_NAME`；依赖需包含 `@langchain/openai`、`@langchain/core`、`zod`、`dotenv`、`mysql2`（智能录入）；JSON Schema 直连示例需 `zod-to-json-schema`；流式终端着色需 `chalk`。MySQL 可用 Docker 本地启动（如设置 `MYSQL_ROOT_PASSWORD`），并用 [MySQL Workbench](https://dev.mysql.com/downloads/workbench/) 等工具连接调试。依赖安装统一使用 **pnpm**。

---

## 一、与上一节的关系

- **`model.withStructuredOutput(schema)`**：非流式下常用，底层按模型能力在 **tool** 与 **output parser** 间选择，保证输出符合 Zod schema。
- **流式场景**：若要打字机效果、尤其是 **tool 调用参数的边生成边展示**，需要直接使用 **output parser**（如 **`JsonOutputToolsParser`**）配合对 **`AIMessageChunk`** 的拼接，而不能指望 `withStructuredOutput` 在流式里逐段吐出完整结构化对象。

本节通过两个实战串联：**智能录入**（结构化 + 入库）与 **流式版 mini cursor**（流式 + tool 参数解析）。

---

## 二、实战一：智能录入（`withStructuredOutput` + MySQL）

- **业务形态**：从表单 / Excel 固定结构录入，升级为给模型**一段自然语言**，由其抽取多人、多字段的结构化数据，再**批量写入数据库**（常见 AI 应用）。
- **建库建表（`create-table.mjs`）**：使用 **`mysql2/promise`** 连接本地 MySQL，创建 database（如 `hello`）、**`friends`** 表（姓名、性别、出生日期、公司、职位、手机、微信等），并可插入一条 demo 数据验证连接。
- **抽取与插入（`smart-import.mjs`）**：
  - 用 **Zod** 定义单条记录 schema，与表字段对齐；再用 **`z.array(friendSchema)`** 表示「文本中可能有多人」。
  - **`const structuredModel = model.withStructuredOutput(friendsArraySchema)`**，prompt 说明如何从一段文本中拆人、缺字段填 **`null`**、日期格式等。
  - 对 **`structuredModel.invoke(prompt)`** 的返回数组做 **`INSERT ... VALUES ?`** 批量插入（`mysql2` 支持二维数组占位），完成 **解析 → 落库**。

---

## 三、`withStructuredOutput` 底层补充：原生 JSON Schema

- 除 **tool**、**output parser** 外，部分模型支持在 **`modelKwargs.response_format`** 中传入 **`type: "json_schema"`**（含 `name`、`strict`、`schema`），让模型按 JSON Schema 直接输出字符串，客户端再 **`JSON.parse`**。
- 可用 **`zodToJsonSchema(zodSchema)`** 将 Zod 转为原生 schema 对象后填入上述字段（示例见 **`structured-json-schema.mjs`**）。
- **日常开发**：仍优先 **`withStructuredOutput`**，由 LangChain 按模型自动选用 **tool / JSON Schema / output parser**；需理解底层机制时再查阅本节与官方文档。

## 四、实战二：流式版 mini cursor

- **问题**：非流式 Agent 长时间无输出，用户不知道模型是否在生成大段 **`write_file`** 内容。
- **非流式循环回顾**：`SystemMessage` / `HumanMessage` → 模型返回带 **`tool_calls`** 的 **`AIMessage`** → 执行 tool → **`ToolMessage`** 写回 **memory** → 直到不再出现 `tool_calls`。
- **流式下的两个核心难点**：

| 难点           | 说明                                                                    | 处理方式                                                                                                                                                                                                                                  |
| -------------- | ----------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 消息不完整     | 流式得到的是 **`AIMessageChunk`**，不能直接当作一轮完整回复写入 history | 在 **`for await (...stream)`** 中用 **`fullAIMessage = fullAIMessage ? fullAIMessage.concat(chunk) : chunk`**，流结束后得到完整 **`AIMessage`**，再 **`history.addMessage(fullAIMessage)`**                                               |
| 工具参数是片段 | **`tool_call_chunks`** 里是增量片段，**不是**随时可用的合法完整 JSON    | 使用 **`JsonOutputToolsParser`**（来自 **`@langchain/core/output_parsers/openai_tools`**），对流中已拼接的 **`fullAIMessage`** 调用 **`parseResult([{ message: fullAIMessage }])`**，得到当前可解析的 tool 参数；解析失败可忽略，继续累积 |

- **`mini-cursor.mjs` 实现要点**：
  - 使用 **`InMemoryChatMessageHistory`** 管理多轮对话。
  - **`model.bindTools(tools).stream(messages)`**；循环内 **concat** 出 **`fullAIMessage`**，同时 **`JsonOutputToolsParser`** 解析。
  - 对 **`write_file`**：用 **`Map`**（key 可为 `toolCall.id` 或 `filePath`）记录 **`args.content` 已打印长度**，每次只 **`stdout.write` 新增子串**，实现代码「打字机」预览。
  - **执行工具**：仍以流结束后的 **`fullAIMessage.tool_calls`** 为准调用 **`tool.invoke`**，避免用未完成的流式解析结果写盘。

## 五、代码要点

| 模块             | 要点                                                                                                                                                                      |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 建表 / demo      | **`mysql2/promise`** → **`createConnection`** → **`CREATE DATABASE` / `CREATE TABLE` / `INSERT`**；`multipleStatements` 按需开启。                                        |
| 智能录入         | **Zod** 单条 + **`z.array`** → **`withStructuredOutput`** → **`invoke`** → 映射为批量 **`VALUES ?`** 插入。                                                               |
| JSON Schema 直连 | **`zodToJsonSchema`** + **`modelKwargs.response_format.json_schema`** → **`invoke`** → **`JSON.parse(res.content)`**。                                                    |
| 流式 Agent       | **`bindTools` + `stream`** → **`AIMessageChunk.concat`** → **`JsonOutputToolsParser.parseResult`** → 增量打印；完整轮次与 tool 执行依赖 **concat 后的 `fullAIMessage`**。 |

---

## 六、总结

- **智能录入**：自然语言 → **`withStructuredOutput`**（Zod）→ 结构化数组 → **MySQL 批量插入**；体现「结构化输出」在业务入库中的典型用法。
- **流式 mini cursor**：在保留 **tool 循环 + memory** 的前提下，用 **`concat` + `JsonOutputToolsParser`** 解决 **chunk 与 `tool_call_chunks`** 问题，实现 **`write_file` 内容流式预览**；真正写文件仍用完整 **`tool_calls`**。
- **输出控制常见两类需求**：**一次性结构化**（优先 `withStructuredOutput`，并了解 JSON Schema 为底层选项之一）；**流式 + tool 参数解析**（手写流式循环与 **`JsonOutputToolsParser`**）。
