# 结构化大模型输出：output parser 还是 tool？ - 总结

---

## 一、为什么需要结构化输出

- 大模型默认输出是自然语言文本；业务常需要 **JSON、XML** 等固定结构，便于程序解析、入库或调用下游接口。
- 若只在 prompt 里写「请返回 JSON」，模型可能返回带 markdown 代码块或多余说明的文本，直接 **JSON.parse** 容易失败；需要对「格式说明 + 解析」做统一封装，即 **Output Parser**，或改用 **Tool 参数** 约束输出。

## 二、Output Parser 思路

- **原理**：在 prompt 中注入**格式说明**（如 parser.**getFormatInstructions()**），要求模型按该格式输出；拿到 **response.content** 后用 parser.**parse(content)** 解析成对象或指定格式。
- **JsonOutputParser**：仅负责解析 JSON，能处理被 markdown 包裹或带前后文字的情况；getFormatInstructions 可为空或简单说明。
- **StructuredOutputParser**：显式定义字段与描述。**fromNamesAndDescriptions({ 字段名: "描述", ... })** 或 **fromZodSchema(zodSchema)**；会生成较长的格式说明拼进 prompt，parse 时按该结构解析并做基本校验。
- **XMLOutputParser**：要求模型输出 XML，getFormatInstructions 给出 XML 约定，parse 得到解析结果；适用于 XML、或非 JSON 的固定格式。

## 三、用 Tool 得到结构化结果

- **做法**：定义一个**仅用于返回结构、不执行逻辑**的 tool（name + description + **schema**，schema 用 zod 描述），**model.bindTools([...])** 后 **invoke**；模型会返回 **tool_calls**，从 **response.tool_calls[0].args** 即可拿到已结构化的参数。
- **优点**：模型在训练时即保证 tool call 参数符合 schema，比「在文本里生成 JSON 再解析」更稳定。
- **适用**：只需一次性拿到结构化数据、且为 JSON 友好结构时，用 Tool 方式通常优于纯 Output Parser。

## 四、withStructuredOutput：首选 API

- **model.withStructuredOutput(zodSchema)**：一次调用即得到结构化对象；内部会判断模型是否支持 tool calls，支持则用 tool 方式，否则走 output parser，无需自己分支。
- **用法**：`const structuredModel = model.withStructuredOutput(scientistSchema); const result = await structuredModel.invoke("介绍一下爱因斯坦");`，result 已是符合 schema 的对象。
- **限制**：底层若用 tool，流式时会在「整份 JSON 生成并校验通过」后才返回，**不是**边生成边输出的真流式；且只适用于可映射为 tool 参数的结构（本质是 JSON），XML/YAML 等仍需用对应 Output Parser。

## 五、流式与 Output Parser / Tool 的取舍

| 场景                            | 推荐方式                                                                                             | 说明                                                                                                                       |
| ------------------------------- | ---------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| 非流式、要 JSON 结构化          | **withStructuredOutput(schema)**                                                                     | 自动选 tool 或 parser，拿到的即结构化对象。                                                                                |
| 流式、边生成边打印文本          | **model.stream(prompt)** + 结束时用 **Output Parser** 解析完整 content                               | 真正逐 chunk 输出；流结束后再 **parser.parse(fullContent)** 得到结构。                                                     |
| 流式、但用 withStructuredOutput | 不推荐                                                                                               | 会等整份结果再返回一个 chunk，无逐字流式效果。                                                                             |
| 流式、且想边收边用 tool 参数    | **model.bindTools(...).stream()**，读 **chunk.tool_call_chunks**；或 **pipe(JsonOutputToolsParser)** | tool_call_chunks 里是参数片段；**JsonOutputToolsParser** 可把片段拼成（可能未完成）的 tool_calls，便于实时展示或部分调用。 |
| 要求 XML / YAML 等非 JSON       | **XMLOutputParser** 等对应 Parser                                                                    | withStructuredOutput 不覆盖这类格式。                                                                                      |

- **JsonOutputToolsParser**：与 **modelWithTool.pipe(parser)** 配合，对流式返回的 tool_call_chunks 做解析，在 chunk 未传完时也能得到「当前已拼好」的 args 对象，适合需要「流式展示 + 实时用参数」的场景。

## 六、代码要点

| 方式                    | 要点                                                                                                                                                                              |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| JsonOutputParser        | **new JsonOutputParser()**；prompt 中可加 **parser.getFormatInstructions()**；**parser.parse(response.content)**。                                                                |
| StructuredOutputParser  | **fromNamesAndDescriptions** 或 **fromZodSchema(zodSchema)**；**getFormatInstructions()** 拼进 prompt；**parse(content)**。                                                       |
| Tool 取结构             | **bindTools([{ name, description, schema }])**，不实现执行函数；**response.tool_calls[0].args**。                                                                                 |
| withStructuredOutput    | **model.withStructuredOutput(zodSchema)**；**invoke** 直接得结构化对象。                                                                                                          |
| 流式 + Parser           | **model.stream(prompt)**，for await 拼 **fullContent**，最后 **parser.parse(fullContent)**。                                                                                      |
| 流式 + tool_call_chunks | **modelWithTool.stream()**，从 **chunk.tool_call_chunks[0].args** 取片段；或 **chain = modelWithTool.pipe(JsonOutputToolsParser)**，**chain.stream()** 得到逐步拼好的 tool call。 |

## 七、总结

- **结构化输出**有两种实现路径：**Output Parser**（在 prompt 里加格式说明 + 解析文本）和 **Tool**（用 tool 的 schema 约束，从 tool_calls[0].args 取结果）；Tool 方式在模型侧更稳。
- **首选**：仅需一次性 JSON 结构化时用 **withStructuredOutput(schema)**，由框架在 tool 与 output parser 间自动选择。
- **仍用 Output Parser 的场景**：流式逐字/逐段打印后再解析、或要求 **XML/YAML** 等非 JSON 格式时，用 **StructuredOutputParser / XMLOutputParser** 等；流式下若还要实时用 tool 参数，可配合 **JsonOutputToolsParser** 解析 tool_call_chunks。
- **依赖**：`@langchain/core`、`@langchain/openai`、`zod`、`dotenv`；环境变量与前面章节一致。
